# B-i read-closure envelope — design + parity harness (implementation brief)

Origin: 2026-06-09. Plan item B-i of
[2026-06-09-cf-cross-scope-architecture-plan.md](2026-06-09-cf-cross-scope-architecture-plan.md).
Normative spec: VTN8.3 (`spec/protocol/v2-turn-network.md`), conformance
CA14.18 (`spec/protocol/cell-authority.md`) — added on this branch. This note
is the wave-2 implementation brief; it is not itself normative.

## Why (one paragraph)

Planned-transcript (cross-scope, mostly movement) envelopes ship the
scope-wide authority slice (~1.7 MB p95 deployed) although commit validation
(`submitShadowCommit` → `validateTranscriptWithCellReader`,
`src/core/shadow-commit-scope.ts` ~295) reads only the transcript-touched
cells. The envelope RPC is 80% of the submit phase, which is 66% of deployed
turn wall. Restricting the envelope authority to the turn's read closure is a
byte-shrinking change with provably unchanged verdicts (VTN8.3 argument).

## Implementation sketch

1. **Closure builder** (new, `src/core/` near the executor): from a planned
   `EffectTranscript`, produce the object/cell id set:
   reads ∪ write pre-images ∪ actor ∪ submitting session; then materialize
   pages via the existing authority-payload path with an explicit id list and
   NO scope expansion (the executor already merges transcript-touched ids at
   `src/core/executor.ts` ~851 — the change is *only those*, dropping the
   scope-wide base). Lineage closure comes free from the lineage-complete
   transfer fill (`src/core/authority-slice.ts` ~630, c95ee22) — verify it
   runs on this path, don't reimplement.
2. **Flag**: `WOO_V2_READ_CLOSURE_ENVELOPE`, read in
   `persistent-object-do.ts` next to `WOO_V2_SLIM_WARM_ENVELOPE` (~1794) and
   threaded as `hooks.readClosureEnvelope`, applied where
   `slimMcpEnvelopeBody` branches in `src/mcp/gateway.ts` (~955): slim path
   stays for same-scope; closure path replaces the full body for
   planned-transcript commits. The `E_SNAPSHOT_REQUIRED` reseed + full-body
   retry (gateway.ts ~975) is unchanged and remains the cold-scope escape.
3. **Repair attempts**: a retry after `missing_state` adds the named cells to
   the closure (the repair loop already threads `repairObjectIds`); it does
   NOT fall back to the scope-wide slice.

## Parity harness (the gate that makes this delegable)

Two layers, both required before the flag ships beyond a lane (VTN8.3):

1. **Corpus parity (fast, curated `npm test`)**: a recorder hook captures
   (commit-scope pre-state snapshot, planned transcript, full-slice authority,
   closure authority) tuples during a cf-local shared-scenario run; a test
   replays each tuple through `submitShadowCommit` twice — relay seeded with
   full slice vs with closure — and asserts identical verdict, reason,
   `mismatched_read_cells`, and post-state hash. Store the corpus as a test
   fixture regenerated by a script, not checked-in bytes that rot.
2. **Lane parity (cf-local + cf-dev)**: run the shared scenario flag-off and
   flag-on; assert identical per-turn verdict streams and final world state;
   with flag-on, the C2 cross-scope byte ceiling (< 256 KB) flips from
   TRACKED to ENFORCED.

Failure semantics to test explicitly (not just happy path):
- a touched cell absent from closure AND scope durable state →
  `missing_state`-class rejection naming the cell (never silent accept);
- a fresher row already at the commit scope is not displaced by a closure
  page (version gate);
- read_version_mismatch repair round-trip works identically under the flag.

## Explicit non-goals

- No change to validation semantics, repair replies, or trust model. A
  version-only (no-pages) attestation envelope was considered and REJECTED —
  it changes the trust model for foreign cells (see plan note).
- No change to the `sessions` envelope field or post-commit delivery.
- Same-scope slim-warm envelopes (already deployed) are untouched.
