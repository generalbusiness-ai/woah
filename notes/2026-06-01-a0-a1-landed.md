# 2026-06-01 — A0 + A1 landed (mobile-heap sequence)

Branch `mobile-heap-a0a1` off `846d0a8`. A0 and A1 are mutually defining: A0 says
what the system is; A1 is the executable check that A0's invariant holds on
multi-node topology. Neither stands alone.

## A0 — Target architecture + coherence invariant (spec)

Added normative **VTN0** to `spec/protocol/v2-turn-network.md`, ahead of VTN1:
- States the target: gossip-routed actor-local **mobile object heap** (the five
  load-bearing claims).
- Defines the **coherence invariant (CI):** for every durable cell there is
  exactly one authority (the commit-scope head); every other materialization is a
  content-addressed, provenance-tagged, read-through projection; no cell has two
  independent write paths; a derived copy is never a write-authority source.
- Records that the shipped DO-per-scope + sparse-gateway profile is a **partial
  realization** (claims 1–2 + fixed-assignment claim 3), and that work proceeds by
  strengthening the CI first, then generalizing assignment into mobility.
- Conformance clause: the CI is machine-checkable on the multi-node topology and
  no authority/cache/fanout/routing change is conformant unless that gate is green.

## A1 — Multi-node coherence gate (test + named script)

- `npm run gate:authority` → the cross-shard `cf-local-walkthrough` (4 shards,
  real DO shapes, two cross-actor sessions). This is the named inner-loop gate
  VTN0 references; required for any authority/cache/fanout/routing change.
- Added the **CI assertion**: after the walkthrough, scan captured
  `woo.commit_rejected.errors` diagnostics (CommitScopeDO logs every rejection
  with the diverged `<object>.<cell>`). This catches CI violations the existing
  step-level walkthrough **masks** — a turn that is rejected, retried, and then
  succeeds passes the step while a derived view silently disagreed with authority.

### A1 immediately found a real masked CI violation on the base
First run (zero-tolerance) FAILED on the current base: `the_outline:leave` by a
guest had its commit rejected with `read version mismatch the_chatroom.subscribers`
(transcript=2 actual=1) and `the_chatroom.session_subscribers`, then retried to
success — so the walkthrough was green while a CI violation occurred every run.
These are the **presence-projection cells** (`subscribers`/`session_subscribers`,
same class as `contents`) still sitting on the commit-validation path. This is
exactly the defect class **A4** (contents/presence-as-projection) retires.

### Ratchet design (so the gate is usable before A4)
The assertion is a **ratchet, not yet zero-tolerance**:
- `KNOWN_CI_DEBT_CELLS = { subscribers, session_subscribers, contents }` — the
  one known pre-A4 class — is allow-listed.
- ANY other diverged cell (a real property/lineage/location authority cell), any
  `post_state_mismatch`, or any `incomplete_transcript` fails the gate
  immediately. Those imply a *second writer*, not a stale projection, and are
  never excusable.
- The allow-list MUST only shrink. **A4's exit criterion is that it empties** —
  at which point this becomes zero-tolerance with no test change.

This stops *new* multiplication today without requiring A4 first, and it converts
A4 from an abstract "fix contents" into a concrete, gated objective: drive
`KNOWN_CI_DEBT_CELLS` to empty.

## Why A0+A1 had to land together
A0 without A1 is aspirational prose that the next fix can quietly violate (as the
base already did). A1 without A0 is a test asserting an invariant nobody declared
normative. Together: VTN0 declares the CI; `gate:authority` enforces it; the
allow-list names the exact remaining debt and ties it to A4.
