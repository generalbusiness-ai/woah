# C4 Load Gate — Measurement Results and CA12.1 Decision Input

Origin: 2026-06-10. Implements the C4 item from
`notes/2026-06-09-cf-cross-scope-architecture-plan.md` (Track C → C4).

## What was built

`scripts/load-cf-dev.ts` (`npm run load:cf-dev`): a standalone workload
harness that boots `wrangler dev` (wrangler.smoke.toml), opens N pooled
guest sessions, issues concurrent chat + movement turns against one room
(the_chatroom / the_deck round-trips), and emits a summary with:

- turn counts (ok / failed / abandoned), classified by root cause
- attempts distribution (histogram, p50, p95)
- turn latency p50/p95 (client-side and server-side from woo.metric stream)
- conflict count and classification (true / false / unknown)
- PASS/FAIL gate against named thresholds

Gate registered in `package.json` as `load:cf-dev` alongside the other slow
lanes (NOT in `npm test`).  When to run: before deploying changes that affect
the room sequencer, the CommitScopeDO, or the `object_live` cell type.

## Conflict classification

**Zero OCC conflicts observed at both N=10 and N=20.**

The workload issued concurrent `say` and `southeast`/`west` turns from N
actors against the same CommitScopeDO (the_chatroom). No
`commit_rejected`/`read_version_mismatch` events were captured in either
the client retry counters or the server `shadow_commit_rejected` metric
stream.

The failures observed were:

1. **Session flakes (A1 pre-existing)**: E_PERM / E_NOSESSION errors when
   an actor issues a verb against a room whose gateway shard no longer
   has an active session for it.  This is the known A1 issue — stale pooled-
   guest sessions linger in the Directory, `primarySessionForActor` picks
   the closed session, `moveto_actor` with `is_primary:false` skips the
   physical move, and subsequent verbs fail E_PERM.  These are NOT OCC
   conflicts; fixing them requires A1 (session lifecycle as first-class
   state).

2. **Cold-start transients (workerd artefact)**: 503 "worker restarted
   mid-request" errors during the first parallel load burst.  The
   serialized-enter warmup and 50ms stagger mitigate most of these.

3. **One server-side repair attempt (N=20)**: attributed to
   `missing_state` in the server metric stream.  Not an OCC conflict.

## CA12.1 Decision Input

**CA12.1 deferral is confirmed correct.** No false conflicts were observed.

CA12.1 (split `object_live` into exact `live:location:<obj>` per-member
cells and `projection:contents:<room>/<member>` cells) is designed to
eliminate false conflicts where two actors moving simultaneously both read
the same coarse `object_live` page version even though their moves touch
DIFFERENT objects (actor A's location cell and actor B's location cell
are independent). Under today's implementation:

- Actor location cells are per-actor (CA3/CA6 actor-anchored movement).
  Two actors moving simultaneously commit at different actor-home DOs
  (each actor's location is owned by their home shard).
- The room's contents projection is updated by fanout, not by a sequenced
  room commit.
- The CommitScopeDO for the_chatroom serializes `say` turns, but `say`
  is currently implemented as a direct_call (not a sequenced room commit),
  so its commit path does NOT go through the chatroom's CommitScopeDO.

**Result**: the current implementation already keeps movement and speech
off the shared room sequencer for the tested workload.  The 0 OCC conflicts
at N=20 confirm that today's coarse `object_live` page does NOT produce
false conflicts under the chat+move mix at this scale.

**CA12.1 deferral condition (from the plan)**:
> "Deferring it is correct only while measurement says contention is low."

The measurement says contention is **zero** — not just low.  The plan's
deferral condition is satisfied at N=10 and N=20.

**Recommendation: DEFER CA12.1.** The data does not justify pulling it
forward.  The next trigger would be:

1. Verbs that genuinely write a shared `contents` or shared `prop` cell
   concurrently (e.g. a room property mutated by every actor per turn).
2. A load test at N=50+ showing OCC conflicts emerging from the sequencer.
3. Any deployment measurement showing `commit_rejected` events attributed
   to `read_version_mismatch` on `object_live` cells.

At that point, re-run `npm run load:cf-dev -- --actors=20` and check the
conflict classification.  If `false > 0`, pull CA12.1 forward.

## Gate thresholds chosen

| Threshold | Value | Rationale |
|---|---|---|
| `occ_abandoned_rate` | 0 | Any OCC-abandoned turn is a sequencer bug. |
| `attempts_p95` | 3 | Allows 1–2 OCC retries; >3 signals retry storm. |
| `latency_p95_ms` | 60,000 | Safety net only (10× observed workerd p95). |

These thresholds are calibrated on the 2026-06-10 workerd-local numbers.
Workerd-local is NOT a prod SLO benchmark — cross-colo RTT and cold-start
costs will make prod numbers higher.  The gate's job is catching COLLAPSE
(retry storms, serialization meltdown), not enforcing prod SLOs.

## Known issues noted but not gated

**A1 session flakes (E_PERM / E_NOSESSION)**: visible at N=10 (10 flakes)
and N=20 (28 flakes).  These are the pre-existing A1 bug.  The gate
excludes them from `occ_abandoned_rate` and notes them in the output.
Once A1 lands, these should disappear and the NOTE should stop appearing.

**N=20 latency increase**: p50 latency increased from ~1500ms (N=10) to
~4800ms (N=20) client-side.  Server-side p50 was ~422ms at N=20 (lower
than N=10's ~800ms), suggesting the extra wall time is queuing in workerd
(single-process local DO with no parallel DO execution).  This is expected
workerd-local behaviour, not a prod signal.
